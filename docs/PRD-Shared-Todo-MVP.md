# PRD - Shared Todo MVP

## 1. 문서 정보
- 문서명: 제품 요구사항 문서 - Shared Todo MVP
- 대상 프로젝트: `YourTodo` Android 앱 + `yourtodo-server`
- Android 기준 브랜치: `codex/shared-todo-mvp`
- 서버 기준 브랜치: `codex/shared-todo-mvp`
- 작성일: 2026-05-09
- 상태: 사용자 정책 결정 반영, TRD 작성 후 구현 착수

## 2. 제품 한 줄 정의
친구에게 할 일을 부담 없이 묶어 보내고, 받은 사람은 선택적으로 수락하며, 보낸 사람은 상대의 진행 상황을 확인할 수 있는 친구 기반 Todo 협업 기능.

## 3. 사용자 문제
친구에게 할 일을 하나씩 보내면 받는 사람은 알림과 요청이 계속 쌓여 피로감을 느낀다. 반대로 보낸 사람은 여러 일을 정리해서 전달하고 싶어도 지금 구조에서는 단건 요청밖에 없다.

또한 받은 사람 입장에서는 친구가 보낸 일이 곧바로 내 할 일에 들어오면 통제감을 잃는다. Todo 앱은 개인의 실행 공간이므로, 친구가 보낸 일은 반드시 수락/거절 과정을 거친 뒤 내 할 일에 들어와야 한다.

이 기능은 관리 도구가 아니라 친구 사이의 요청과 수행을 돕는 기능이어야 한다. `감시`, `관리`, `지시` 느낌을 피하고 `주고받은 일`, `진행 상황`, `함께 확인`의 톤을 유지한다.

## 4. 목표
- 친구에게 단건 또는 묶음으로 할 일을 보낼 수 있다.
- 묶음 전송은 여러 할 일을 한 번에 보내고 알림도 한 번만 발생시켜 피로도를 낮춘다.
- 받은 사람은 묶음 안의 할 일을 체크박스로 골라 항목별로 수락/거절할 수 있다.
- 수락한 항목만 내 Todo에 들어오며, 내가 만든 할 일과 구분된다.
- 받은 사람은 친구가 보낸 본문을 수정할 수 없고, 본인 리마인더만 수정할 수 있다.
- 보낸 사람은 친구별로 내가 준 일, 받은 일, 진척도를 확인할 수 있다.
- 보낸 사람은 아직 완료되지 않은 보낸 일을 취소할 수 있고, 취소 알림이 간다.
- 친구 삭제 액션은 기존 점세개보다 명확한 휴지통 아이콘 기반 UI로 개선한다.

## 5. 비목표
- FCM 실제 발송 구현. 이번 단계에서는 이벤트/딥링크 계약을 고정하고 실제 푸시 발송은 후속 PR로 분리할 수 있다.
- 항목별 진행 메모, 댓글, 첨부 파일.
- 보낸 뒤 본문 수정.
- 받은 사람의 본문 수정.
- 복잡한 실시간 동기화. 앱 재진입, 화면 refresh, API pull 기반으로 최신 상태를 보장한다.
- 서버에 draft 묶음을 저장하는 기능. MVP에서는 Android 로컬 draft 후 최종 전송 시 서버에 리스트로 보낸다.

## 6. 핵심 원칙
### 6.1 동의 먼저
친구가 보낸 할 일은 바로 내 Todo가 아니다. 항상 `PENDING_ACCEPTANCE` 상태로 도착하고, 받은 사람이 수락한 항목만 내 Todo에 들어온다.

### 6.2 묶음은 알림 단위, 항목은 결정 단위
`AssignmentBundle`은 전송, 알림, 딥링크, 요약 단위다. 실제 수락/거절/진행/완료는 묶음 안의 `AssignedTodo` 항목별로 발생한다.

### 6.3 수신자의 통제감
받은 사람은 묶음 안에서 받을 일을 체크박스로 고른다. 전체 수락은 별도 특수 버튼이 아니라 `전체 선택` 후 `n개 수락`으로 해결한다.

### 6.4 본문은 약속, 리마인더는 개인 설정
제목, 내용, 마감일, 우선순위, 카테고리, 체크리스트 제목은 보낸 사람이 정한 약속이다. 받은 사람은 이를 수정하지 않는다. 단, 리마인더는 개인 실행 방식이므로 받은 사람이 본인 기준으로 수정할 수 있다.

### 6.5 푸시는 정합성 원천이 아니다
푸시는 사용자가 놓친 사건을 알려주는 장치다. 푸시 실패나 지연이 데이터 정합성을 깨면 안 된다. 앱을 열면 서버 API로 최신 상태를 확인할 수 있어야 한다.

## 7. 주요 사용자 플로우
### 7.1 친구 작업 허브
1. 사용자가 친구 탭에서 친구를 누른다.
2. 친구별 bottom sheet가 열린다.
3. 사용자는 `할 일 보내기`, `내가 준 일`, `{nickname}에게 받은 일`, `진행 요약`을 볼 수 있다.
4. 친구 삭제는 row의 휴지통 아이콘 또는 sheet의 명확한 danger action으로 접근한다.

권장 문구:
- Sheet 제목: `{nickname}와 할 일`
- 보조 문구: `서로 주고받은 할 일을 확인해요.`
- 액션: `할 일 보내기`, `내가 준 일`, `{nickname}에게 받은 일`, `진행 요약`
- `진행 요약`은 별도 무거운 관리 화면이 아니라 `내가 준 일`의 상태 요약과 최근 업데이트 진입점으로 둔다.

### 7.2 즉시 보내기
1. 사용자가 친구 작업 허브에서 `할 일 보내기`를 누른다.
2. 상단 전송 방식에서 `즉시 보내기`를 선택한다.
3. 일반 Todo 입력과 거의 같은 필드를 입력한다. 리마인더는 제외된다.
4. `바로 보내기`를 누른다.
5. 서버에는 1개 항목을 가진 bundle이 생성된다.
6. 받은 사람에게 bundle 기준 알림 이벤트가 1회 생성된다.

### 7.3 묶어서 보내기
1. 사용자가 전송 방식에서 `묶어서 보내기`를 선택한다.
2. 할 일을 작성하고 `묶음에 추가`를 누른다.
3. 항목은 Android 로컬 draft 묶음에 저장되며 받는 사람에게 보이지 않는다.
4. 사용자는 여러 항목을 추가, 수정, 삭제할 수 있다.
5. `검토 후 보내기`를 누르면 전송 전 검토 화면으로 이동한다.
6. 검토 화면에서 `{n}개 보내기`를 누르면 서버에 item list가 전송된다.
7. 서버는 all-or-nothing으로 bundle을 생성한다.
8. 받은 사람에게 알림 이벤트는 bundle 기준 1회만 생성된다.

권장 문구:
- `묶음에 추가`
- `검토 후 보내기`
- `{nickname}에게 보낼 묶음`
- `{n}개가 한 번에 전송돼요.`

### 7.4 묶음 수락/거절
1. 받은 사람이 푸시를 누르거나 앱 내 받은 요청을 연다.
2. 묶음 수락/거절 화면으로 이동한다.
3. 화면에는 `전체 선택`, 선택 개수, 항목별 체크박스가 표시된다.
4. 사용자는 처리할 항목을 선택한다.
5. `n개 수락` 또는 `n개 거절`을 누른다.
6. 수락된 항목만 내 Todo에 들어온다.
7. 거절된 항목은 내 Todo에 들어오지 않는다.
8. 처리되지 않은 항목이 남아 있으면 이후 재진입 시 pending 상태로 남는다.

권장 문구:
- `{nickname}가 할 일 {n}개를 보냈어요`
- `받을 할 일을 선택해주세요.`
- `전체 선택`
- `선택 {selected}/{total}`
- `{selected}개 수락`
- `{selected}개 거절`

### 7.5 받은 할 일 수행
1. 수락한 항목은 Todo 목록의 `전체`와 `받은 일` 필터에 표시된다.
2. 받은 할 일에는 `친구가 보낸 일` 라벨과 보낸 사람 닉네임이 표시된다.
3. 상세 화면에서 제목, 내용, 마감일, 체크리스트 제목은 읽기 전용이다.
4. 사용자는 시작, 체크리스트 완료, 완료 처리를 할 수 있다.
5. 사용자는 본인 리마인더만 추가, 수정, 삭제할 수 있다.
6. 체크리스트가 있으면 완료 항목 수로 진척도를 계산한다.
7. 체크리스트가 없으면 상태 기반 진척도를 사용한다.

### 7.6 받은 일 삭제
1. 받은 사람은 수락 전 요청을 거절할 수 있다.
2. 받은 사람은 수락 후 받은 일을 삭제할 수 있다.
3. 삭제는 물리 삭제가 아니라 거절 계열 종료 상태로 처리한다.
4. 삭제된 항목은 기본 Todo 목록과 `받은 일` 필터에서 사라진다.
5. 받은 사람 화면에서는 `삭제됨` 또는 `받은 일에서 삭제됨`으로 안내한다.
6. 보낸 사람 화면에서는 MVP에서 `받지 않음`으로 부드럽게 표시한다.

권장 확인 문구:
- 제목: `이 받은 일을 삭제할까요?`
- 설명: `내 할 일 목록에서 사라지고, 보낸 사람에게는 거절된 것으로 표시돼요.`
- 버튼: `취소`, `삭제`

### 7.7 보낸 일 취소
1. 보낸 사람은 내가 준 일 상세에서 아직 완료되지 않은 항목을 취소할 수 있다.
2. 완료된 항목은 취소할 수 없다.
3. 취소하면 받은 사람에게 알림 이벤트가 생성된다.
4. 받은 사람 쪽에서는 더 이상 수락, 진행, 완료, 리마인더 수정을 할 수 없다.
5. 수락 전 취소된 항목은 요청 화면에서 `취소됨`으로 보이고 action이 비활성화된다.
6. 수락 후 취소된 항목은 기본 Todo 목록에서는 제거하고, 받은 일 history 또는 상세 재진입 시 `보낸 사람이 이 일을 취소했어요.` 안내를 보여준다.
7. 기록은 삭제하지 않고 `CANCELED`로 남긴다.

권장 확인 문구:
- 제목: `보낸 일을 취소할까요?`
- 설명: `친구에게 취소 알림이 전달돼요.`
- 버튼: `아니요`, `취소하기`

## 8. Todo 목록 필터
Todo 목록에는 최소 다음 필터를 추가한다.

```text
전체
내 할 일
받은 일
```

- `전체`: 내가 만든 일 + 수락했고 삭제/취소되지 않은 받은 일
- `내 할 일`: 내가 직접 만든 일
- `받은 일`: 친구가 보낸 일 중 수락했고 삭제/취소되지 않은 일
- Android는 기존 개인 Todo sync 결과와 received assigned todo feed를 앱에서 합성해 Todo 목록을 구성한다.
- 서버의 기존 `/api/sync/todos`는 개인 Todo만 반환하고, 친구가 준 받은 일은 `/api/assigned-todos/received?status=active`에서 별도로 받는다.
- `PENDING_ACCEPTANCE`: Todo 목록이 아니라 받은 요청/묶음 수락 화면에서 표시
- `REJECTED`, `CANCELED`: 기본 Todo 목록에서 제외하고, 친구별 기록/보낸 일 화면에서만 상태로 표시

## 9. 상태 정책
### 9.1 항목 상태
```text
PENDING_ACCEPTANCE
ACCEPTED
IN_PROGRESS
DONE
REJECTED
CANCELED
```

### 9.2 묶음 집계 상태
```text
DRAFT
SENT
PARTIALLY_DECIDED
ACCEPTED
REJECTED
DECIDED_PARTIAL
CANCELED
```

집계 규칙:
- 모든 항목 pending: `SENT`
- pending과 non-pending이 섞임: `PARTIALLY_DECIDED`
- 모든 항목이 accepted 계열이면 `ACCEPTED`. accepted 계열은 `ACCEPTED`, `IN_PROGRESS`, `DONE`이다.
- 모든 항목 rejected: `REJECTED`
- 모든 항목 canceled: `CANCELED`
- pending 없이 accepted 계열, rejected, canceled가 섞임: `DECIDED_PARTIAL`
- `DONE + REJECTED + CANCELED`처럼 모든 항목이 terminal이어도 결과가 섞이면 `DECIDED_PARTIAL`

사용자 표시 문구:
- `SENT`: `응답 대기`
- `PARTIALLY_DECIDED`: `일부 처리됨`
- `ACCEPTED`: `수락됨`
- `REJECTED`: `받지 않음`
- `DECIDED_PARTIAL`: `일부 수락됨`
- `CANCELED`: `취소됨`

항목 표시 문구:
- `PENDING_ACCEPTANCE`: `응답 대기`
- `ACCEPTED`: `수락됨`
- `IN_PROGRESS`: `진행 중`
- `DONE`: `완료`
- `REJECTED` + `REJECTED_BY_RECEIVER`: `받지 않음`
- `REJECTED` + `DELETED_BY_RECEIVER`: `삭제됨`
- `CANCELED`: `취소됨`

### 9.3 진척도
- 체크리스트가 있으면 `완료 항목 / 전체 항목`으로 계산한다.
- 체크리스트가 없으면 상태 기반이다.
  - `ACCEPTED`: 0%
  - `IN_PROGRESS`: 50%
  - `DONE`: 100%
- bundle progress percent는 수락된 항목만 분모에 포함한다. 거절하거나 아직 pending인 항목 때문에 진척도가 낮아지면 안 된다.

## 10. 알림/딥링크 정책
이번 기능 PR에서 실제 FCM 발송까지 포함하지 않더라도, 서버 이벤트 타입과 Android 딥링크 목적지는 고정한다.

이벤트:
```text
ASSIGNMENT_BUNDLE_RECEIVED
ASSIGNMENT_BUNDLE_PARTIALLY_DECIDED
ASSIGNMENT_BUNDLE_FULLY_DECIDED
ASSIGNED_TODO_COMPLETED
ASSIGNED_TODO_REOPENED
ASSIGNED_TODO_CANCELED
```

딥링크:
- `ASSIGNMENT_BUNDLE_RECEIVED`: 묶음 수락/거절 화면
- `ASSIGNMENT_BUNDLE_PARTIALLY_DECIDED`: 내가 준 일 bundle 상세
- `ASSIGNMENT_BUNDLE_FULLY_DECIDED`: 내가 준 일 bundle 상세
- `ASSIGNED_TODO_COMPLETED`: 내가 준 일 상세
- `ASSIGNED_TODO_REOPENED`: 내가 준 일 상세
- `ASSIGNED_TODO_CANCELED`: 취소된 받은 일 안내 또는 받은 일 상세

푸시 피로도 정책:
- bundle 전송은 항목 수와 관계없이 수신자 알림 1회.
- 항목별 수락/거절 이벤트는 서버 이벤트로 남기되, 푸시는 bundle 요약 중심으로 보낸다.
- 체크리스트 개별 변경은 푸시하지 않는다.

## 11. 수용 기준
- 친구를 누르면 친구별 작업 bottom sheet가 열린다.
- 친구 삭제 액션은 휴지통 아이콘 또는 명확한 danger action으로 노출된다.
- 할 일 보내기 화면에서 `즉시 보내기`와 `묶어서 보내기`를 선택할 수 있다.
- 친구에게 보낼 할 일 입력에는 리마인더가 없다.
- 묶어서 보내기 draft는 최종 전송 전까지 받는 사람에게 보이지 않는다.
- 묶음 전송 시 서버는 item list를 받고, 수신자 알림 이벤트는 1회만 생성된다.
- 묶음 수락 화면에서 항목별 체크박스, 전체 선택, 선택 개수가 표시된다.
- 선택 항목이 0개면 수락/거절 버튼은 비활성화된다.
- 선택 항목 수에 맞춰 `n개 수락`, `n개 거절` 문구가 표시된다.
- 수락한 항목만 내 Todo에 생성된다.
- 거절한 항목은 내 Todo에 생성되지 않는다.
- 일부 처리 후 재진입하면 pending 항목만 선택 가능하다.
- 받은 할 일 상세에서 본문 필드는 읽기 전용이다.
- 받은 사람은 본인 리마인더만 수정할 수 있다.
- 받은 일을 삭제하면 거절 계열 상태로 처리되고 기본 Todo 목록에서 사라진다.
- 보낸 사람은 완료되지 않은 보낸 일을 취소할 수 있다.
- 취소된 항목은 받은 사람이 더 이상 처리할 수 없다.
- 보낸 사람은 친구별로 내가 준 일, 받은 일, 진행 상황을 확인할 수 있다.
- 권한 없는 사용자는 bundle/item을 조회하거나 처리할 수 없다.

## 12. 성공 지표
- 묶음 전송 사용 시 항목 수 대비 알림 수가 줄어든다.
- 받은 사용자가 묶음에서 일부 항목만 수락할 수 있어 거절 부담이 줄어든다.
- 수락한 받은 일이 내 Todo에서 명확히 구분된다.
- 보낸 사람이 친구별 진행 상황을 확인할 수 있다.
- 사용자가 친구 삭제/보낸 일 취소 같은 위험 액션을 쉽게 발견하되 실수로 실행하지 않는다.

## 13. 오픈 리스크
- 항목별 수락/거절은 상태 조합이 많아져 테스트가 늘어난다.
- 배치 처리 중 네트워크 타임아웃이 발생하면 idempotency가 없을 경우 중복 Todo가 생길 수 있다.
- 친구가 보낸 일을 삭제했을 때 보낸 사람에게 `거절됨`으로 보이는 문구가 일부 사용자에게 차갑게 느껴질 수 있다.
- 푸시 실제 구현이 후속으로 밀리면 사용자는 앱 재진입 전까지 요청을 늦게 볼 수 있다.
